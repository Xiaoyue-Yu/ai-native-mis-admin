# Product Workflows - 系统管理模块需求分析工作流

## 工作流目标

本工作流用于指导 AI 按照标准步骤完成系统管理模块的需求分析、文档拆分和前端原型生成。

该工作流的核心原则是：分阶段推进、单阶段聚焦、阶段间强制停顿、等待人类确认后再继续。

AI 不得一次性生成所有内容，不得在需求尚未澄清时直接生成完整文档或 HTML 原型。

## 适用范围

适用于以下系统管理模块的产品设计工作：

- 用户管理
- 角色管理
- 权限管理
- 部门管理
- 菜单管理
- 数据字典
- 系统配置
- 操作日志
- 登录认证
- 其他后台管理模块

## 总体执行原则

### 必须遵守

- 必须严格按阶段执行。
- 每次只处理当前阶段的任务。
- 每个阶段完成后，必须停顿并等待人类确认。
- 未获得人类明确确认，不得进入下一阶段。
- 必须优先使用指定 skill。
- 必须将需求分析拆分为多个可控上下文。
- 必须保留已确认结论和待确认问题。
- 必须避免一次性生成过长内容导致上下文混乱。

### 禁止行为

- 禁止跳过需求澄清直接写需求文档。
- 禁止阶段 1 未完成就进入阶段 2。
- 禁止阶段 2 未确认就进入阶段 3。
- 禁止阶段 3 未确认就进入阶段 4。
- 禁止一次性输出系统概述、非功能需求、业务规则和 HTML 原型。
- 禁止擅自补充会影响核心设计的业务规则。
- 禁止在用户没有确认时假设需求已经闭环。

## 阶段总览

| 阶段 | 名称 | 使用 Skill | 主要产物 | 是否需要人类确认 |
|------|------|------------|----------|------------------|
| 阶段 1 | 需求澄清 | requirement-clarifier | 已确认需求结论与待确认问题 | 是 |
| 阶段 2 | 系统概述与功能需求 | 无强制 skill | 《系统概述与功能需求.md》 | 是 |
| 阶段 3 | 非功能性需求与业务规则 | 无强制 skill | 《非功能性需求与业务规则.md》 | 是 |
| 阶段 4 | HTML 原型页面 | frontend-design | HTML 原型页面 | 是 |

## 阶段 1：需求澄清

### 目标

使用 `requirement-clarifier` 技能，与用户进行连续对话，完成系统管理模块的需求澄清。

本阶段只做需求澄清，不生成正式需求文档，不生成原型页面。

### AI 必须执行

1. 启用 `.codex/skills/requirement-clarifier.md` 中定义的角色和流程。
2. 复述用户提出的模块需求。
3. 主动识别缺失的业务规则、异常分支、边界条件、权限规则和数据范围。
4. 按主题分组向用户连续追问。
5. 每轮追问后整理：
   - 已确认事项
   - 待确认事项
   - 下一轮问题
6. 直到需求逻辑基本闭环。

### 阶段 1 输出

阶段 1 的最终输出应包括：

```markdown
# 阶段 1 输出：需求澄清结果

## 1. 已确认需求结论

## 2. 已确认业务规则

## 3. 已确认异常分支

## 4. 已确认权限与数据范围

## 5. 待后续确认事项

## 6. 是否可以进入阶段 2

请确认是否进入阶段 2：生成《系统概述与功能需求.md》。
```

### 阶段 1 停顿规则

完成阶段 1 后，AI 必须停止。

必须等待用户明确回复类似以下内容后，才能进入阶段 2：

- “确认，进入阶段 2”
- “可以进入阶段 2”
- “继续生成系统概述与功能需求”

如果用户没有明确确认，AI 不得继续。

## 阶段 2：生成《系统概述与功能需求.md》

### 目标

根据阶段 1 已确认的需求澄清结果，生成正式文档《系统概述与功能需求.md》。

本阶段只生成系统概述和功能需求，不写非功能性需求，不写完整业务规则细节，不生成 HTML 原型。

### AI 必须执行

1. 只使用阶段 1 已确认的信息。
2. 不擅自新增关键业务规则。
3. 将系统目标、范围、用户角色、模块边界、功能清单、页面结构和功能说明整理成文档。
4. 对仍未确认但不阻塞功能描述的事项，标记为“待确认”。
5. 将内容组织为清晰的 Markdown 文档。

### 推荐文档结构

```markdown
# 系统概述与功能需求

## 1. 系统背景

## 2. 建设目标

## 3. 适用角色

## 4. 系统范围

### 4.1 包含范围

### 4.2 不包含范围

## 5. 模块清单

## 6. 页面与功能结构

## 7. 功能需求

### 7.1 [模块名称]

#### 7.1.1 功能说明

#### 7.1.2 查询条件

#### 7.1.3 列表字段

#### 7.1.4 新增功能

#### 7.1.5 编辑功能

#### 7.1.6 删除功能

#### 7.1.7 详情功能

#### 7.1.8 批量操作

#### 7.1.9 导入导出

## 8. 待确认事项
```

### 阶段 2 输出

阶段 2 的最终输出必须包含：

```markdown
# 阶段 2 输出完成

已生成：《系统概述与功能需求.md》

请确认文档内容是否通过。
确认后，我将进入阶段 3：生成《非功能性需求与业务规则.md》。
```

### 阶段 2 停顿规则

完成阶段 2 后，AI 必须停止。

必须等待用户明确回复类似以下内容后，才能进入阶段 3：

- “确认，进入阶段 3”
- “文档通过，继续”
- “继续生成非功能性需求与业务规则”

如果用户要求修改阶段 2 文档，AI 必须先修改阶段 2 文档，再次等待确认。

## 阶段 3：生成《非功能性需求与业务规则.md》

### 目标

根据阶段 1 的澄清结果和阶段 2 的功能需求文档，生成《非功能性需求与业务规则.md》。

本阶段只处理非功能性需求、业务规则、异常分支、权限规则、审计规则、数据规则等内容，不生成 HTML 原型。

### AI 必须执行

1. 引用阶段 1 已确认的业务规则和异常分支。
2. 与阶段 2 的功能需求保持一致。
3. 明确权限控制、数据范围、状态流转、异常处理、日志审计、性能、安全、可用性等规则。
4. 将规则写成可开发、可测试的条目。
5. 对仍未确认的非阻塞项标记为“待确认”。

### 推荐文档结构

```markdown
# 非功能性需求与业务规则

## 1. 业务规则总览

## 2. 权限控制规则

## 3. 数据范围规则

## 4. 状态流转规则

## 5. 字段校验规则

## 6. 异常分支与系统处理

| 场景 | 触发条件 | 系统处理 | 用户提示 |
|------|----------|----------|----------|

## 7. 删除、停用与关联影响

## 8. 操作日志与审计规则

## 9. 安全性需求

## 10. 性能需求

## 11. 可用性与兼容性需求

## 12. 数据初始化与迁移规则

## 13. 待确认事项
```

### 阶段 3 输出

阶段 3 的最终输出必须包含：

```markdown
# 阶段 3 输出完成

已生成：《非功能性需求与业务规则.md》

请确认文档内容是否通过。
确认后，我将进入阶段 4：使用 frontend-design 技能生成 HTML 原型页面。
```

### 阶段 3 停顿规则

完成阶段 3 后，AI 必须停止。

必须等待用户明确回复类似以下内容后，才能进入阶段 4：

- “确认，进入阶段 4”
- “文档通过，继续原型”
- “继续生成 HTML 原型”

如果用户要求修改阶段 3 文档，AI 必须先修改阶段 3 文档，再次等待确认。

## 阶段 4：生成 HTML 原型页面

### 目标

使用 `frontend-design` 技能，根据以下文档生成 HTML 高保真前端原型页面：

- 《系统概述与功能需求.md》
- 《非功能性需求与业务规则.md》

本阶段只生成 HTML 原型页面，不重新发散需求，不重写前两阶段文档。

### AI 必须执行

1. 启用 `.codex/skills/frontend-design.md` 中定义的角色和流程。
2. 严格读取阶段 2 和阶段 3 的文档。
3. 提取所有页面、字段、筛选项、表格列、按钮、弹窗、状态、异常提示和交互。
4. 使用 HTML5 + Tailwind CSS CDN + 原生 JavaScript 生成可运行原型。
5. 确保 PRD 中的字段和操作按钮不遗漏。
6. 在输出后提供覆盖自检清单。

### 阶段 4 输出

阶段 4 的最终输出应包括：

```markdown
# 阶段 4 输出完成

已生成 HTML 原型页面。

## 原型覆盖自检

- 页面：...
- 字段：...
- 操作按钮：...
- 弹窗：...
- 状态与提示：...
- 交互示意：...
```

### 阶段 4 停顿规则

完成阶段 4 后，AI 必须停止。

如果用户要求修改原型，AI 只能基于已确认的阶段 2 和阶段 3 文档修改原型。

如果用户提出新增需求或修改核心业务规则，AI 必须提示用户回到对应阶段重新确认。

## 阶段切换确认语

AI 必须在每个阶段结束时使用明确的确认语：

```markdown
当前阶段已完成。我会先停在这里，等待你确认。

请回复“确认，进入阶段 X”，我再继续下一阶段。
```

不得在同一条回复中自动进入下一阶段。

## 上下文控制规则

为了避免上下文崩溃，AI 必须遵守：

- 每轮只处理一个阶段。
- 每个阶段只输出对应产物。
- 长文档应结构化输出，避免无意义重复。
- 如果内容过长，应分模块输出，并在每段结束后等待确认。
- 不得同时生成需求澄清、功能需求、业务规则和 HTML 原型。
- 不得把未确认的问题混入正式结论。
- 所有待确认事项必须集中列出。

## 异常处理

### 用户要求跳过阶段

如果用户要求跳过需求澄清直接生成文档，AI 应回复：

```markdown
该工作流要求先完成阶段 1：需求澄清。否则后续文档和原型容易遗漏业务规则、异常分支和权限边界。

我将先使用 requirement-clarifier 技能完成需求澄清。
```

### 用户在后续阶段修改核心需求

如果用户在阶段 2、阶段 3 或阶段 4 中提出会影响业务规则、字段、权限或流程的修改，AI 应回复：

```markdown
这个修改会影响已确认的核心需求，需要回到阶段 1 或对应文档阶段重新确认。

我建议先更新需求澄清结论，再继续后续产物。
```

### 用户只修改表达或排版

如果用户只要求调整文档表达、标题、格式或页面视觉样式，AI 可以在当前阶段内修改，不需要回退到阶段 1。

## 最小完成标准

该工作流完成时，至少应产出：

1. 需求澄清结果
2. 《系统概述与功能需求.md》
3. 《非功能性需求与业务规则.md》
4. HTML 原型页面
5. 原型覆盖自检清单

所有产物必须来源于已确认需求，不得来源于 AI 的无依据发散。

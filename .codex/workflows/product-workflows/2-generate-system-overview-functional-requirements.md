# 2 - Generate System Overview And Functional Requirements

## 阶段目标

根据阶段 1 已确认的需求澄清结果，生成《系统概述与功能需求.md》。

本阶段只生成系统概述与功能需求，不写完整非功能需求，不生成 HTML 原型。

## 可参考资料

允许参考：

- 阶段 1 需求澄清结果
- `docs/system-management-srs.md`
- `docs/system-management-use-case.puml`
- `docs/user-management-activity.puml`

参考原则：

- 阶段 1 已确认内容优先。
- `docs/system-management-srs.md` 可用于补充文档结构、功能域、角色和需求追踪方式。
- 如果 SRS 中存在“当前实现状态”，应区分“已实现”“部分实现”“待实现”。

## 必须执行

1. 只基于已确认需求编写正式文档。
2. 不擅自新增关键业务规则。
3. 清晰描述系统背景、目标、范围、角色、模块和功能。
4. 每个功能必须包含查询、列表、新增、编辑、删除、详情、批量、导入导出等适用项。
5. 未确认但不阻塞功能描述的内容放入“待确认事项”。

## 推荐文档结构

```markdown
# 系统概述与功能需求

## 1. 系统背景

## 2. 建设目标

## 3. 适用角色

## 4. 系统范围

### 4.1 包含范围

### 4.2 不包含范围

## 5. 模块清单

## 6. 页面与功能结构

## 7. 功能需求

### 7.1 [模块名称]

#### 7.1.1 功能说明

#### 7.1.2 查询条件

#### 7.1.3 列表字段

#### 7.1.4 新增功能

#### 7.1.5 编辑功能

#### 7.1.6 删除功能

#### 7.1.7 详情功能

#### 7.1.8 批量操作

#### 7.1.9 导入导出

## 8. 待确认事项
```


## 停顿规则

阶段 2 完成后必须停止。

如果用户要求修改阶段 2 文档，必须先修改并再次等待确认。
